# AGENTS.md

Cross-engine benchmark harness for [`oteldb`](../oteldb). It stands up a
docker-compose matrix of observability engines, ingests one canonical dataset
into all of them via a fan-out, replays a shared query suite against each, and
reports query latency alongside storage/memory footprint.

Read `README.md` first — it explains the design and the fairness model in depth.
This file is the operational summary for working in the repo.

## What this repo is

- A single Go CLI, **`benchctl`** (`cmd/benchctl`), drives everything. There are
  no bash/`yq`/`awk` pipelines anymore — the old shell scripts were replaced by
  Go in `internal/bench/`.
- The benchmark matrix (systems × signals × query API × language × storage) lives
  in **`systems.yml`** — the single source of truth every command reads.
- Three signals/lanes: **metrics** (PromQL), **logs** (LogQL), **traces**
  (TraceQL). `oteldb` is benched in three configs from one binary: embedded
  `file` backend (`oteldb`), ClickHouse backend (`oteldb-ch`), and shared S3 via
  go-faster/fs (`oteldb-s3`).

## Build / lint / test

```bash
go build ./...      # build everything
go vet ./...        # vet
gofmt -l .          # must print nothing — CI fails on unformatted files
```

CI (`.github/workflows/ci.yml`) runs exactly: `gofmt -l`, `go build ./...`,
`go vet ./...`, plus a `docker compose ... config --quiet` validation of the
merged compose files. **There are no Go unit tests** — verification is the build,
vet, gofmt, and the compose-config check. Run all four locally before pushing.

Validate the merged compose stack the way CI does:

```bash
docker compose -f compose/base.yml -f compose/shared.yml \
  -f compose/metrics.yml -f compose/logs.yml -f compose/traces.yml \
  --profile metrics --profile logs --profile traces config --quiet
```

## Running the benchmark

Requires `docker` (+ compose v2) and Go ≥ 1.24. Copy `.env.example` to `.env`
first. `benchctl` finds the repo root by walking up to the dir containing
`systems.yml`, so it works from any subdir.

```bash
go run ./cmd/benchctl bench-all          # whole matrix → results/REPORT.md
go run ./cmd/benchctl bench metrics      # one lane: up→ingest→settle→query→check→collect
go run ./cmd/benchctl down               # stop + wipe volumes

# à la carte steps:
go run ./cmd/benchctl up metrics
go run ./cmd/benchctl ingest metrics
go run ./cmd/benchctl query metrics 20   # 20 runs/query after warmup
go run ./cmd/benchctl collect
go run ./cmd/benchctl report
```

The `task` wrapper (`Taskfile.yml`) is just convenience over the same commands
(`task bench:all`, `task bench LANE=metrics`, `task down`).

**Fast feedback:** `--only <csv>` (or `BENCH_ONLY` in `.env`) restricts a run to
a subset of engines — it brings up only those engines plus the lane's ingest
driver and queries/collects/reports just them:

```bash
go run ./cmd/benchctl --only oteldb bench-all
go run ./cmd/benchctl --only oteldb,oteldb-ch bench-all
```

## Layout

```
systems.yml          the matrix; addr is the published 127.0.0.1 port
cmd/benchctl/        CLI entrypoint (arg parsing, command dispatch)
internal/bench/      the engine — one file per concern:
  config.go            Env + .env loading, systems.yml parsing, --only filtering
  compose.go           docker compose orchestration (up/down)
  ingest.go            fan-out ingest (otelbench from ../oteldb; telemetrygen for traces)
  query.go             replay query suites, record latency percentiles
  check.go             verify query results vs a reference count
  collect.go           on-disk (fs bucket / named volume) + RSS footprint
  report.go            render results/REPORT.md
  suite.go             query-suite (queries/*.yml) loading
compose/             base + shared + per-lane (metrics/logs/traces) stacks
config/              one dir per engine; otelcol/ holds the fan-out configs
queries/             PromQL/LogQL/TraceQL suites (+ native/ dialect overrides)
results/             per-system CSVs, raw/ latencies, REPORT.md (generated)
```

## Conventions

- **Go only**, module `github.com/oteldb/benchmark`, Go 1.24. The single
  third-party dependency is `gopkg.in/yaml.v3`; everything else is stdlib.
  Errors use stdlib `fmt.Errorf("...: %w", err)` — this repo does **not** use
  go-faster/errors.
- Keep `systems.yml` order intentional: report columns follow file order, and the
  parser (`LoadSystems`) preserves it via `yaml.Node` rather than maps. When
  adding an engine, add it to `systems.yml`, the right `compose/*.yml`, a
  `config/<engine>/` dir, the otelcol fan-out, and a Grafana datasource.
- Ports in `systems.yml` are the host-published `127.0.0.1` ports; the in-network
  fan-out collectors and Grafana datasources use docker-internal hostnames.
- Generated/derived files: `config/vmagent/scrape.yml` is regenerated by
  `benchctl up` from `BENCH_NODES`; `results/REPORT.md` and the CSVs under
  `results/` are produced by `report`/`query`/`collect` — don't hand-edit them.
- Keep the package doc comments accurate — every `internal/bench` file leads with
  a comment describing its role; update it when behavior changes.

## Tuning knobs (`.env`)

`BENCH_NODES` (metrics cardinality — synthetic node_exporter hosts), `BENCH_RUNS`
(runs/query after warmup), `BENCH_CPUS`/`BENCH_MEMLIMIT` (resource caps for
oteldb), `BENCH_ONLY` (subset of engines), `LOGHUB_DIR` (real log data for the
LogQL suite), `OTELDB_SRC` (sibling oteldb checkout used to run `otelbench`
during ingest), and the `*_IMAGE` pins.
